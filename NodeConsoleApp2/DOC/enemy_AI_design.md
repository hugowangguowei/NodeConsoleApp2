# 敌人 AI 重构设计文档（MVP）

## 1. 文档目标

本文档用于将当前可实施的敌人 AI 重构方案固化为正式设计方案，作为后续实现与验收依据。

本方案遵循以下原则：

- 优先满足当前测试需求
- 先实现简单、稳定、可验证的敌人交互
- 不做过度设计
- 保持高内聚、低耦合
- 行为层与技能层解耦
- 严格按数据配置驱动，不做隐式兜底
- 为后续扩展预留清晰接口

---

## 2. 当前重构目标

当前阶段的敌人数量不多，测试需求也较基础，因此本次重构目标不是实现复杂 AI，而是建立一套：

1. 能稳定运行
2. 符合现有战斗设计
3. 能支撑后续扩展
4. 不继续在 `CoreEngine` 中堆叠临时逻辑

本次重构重点解决以下问题：

1. 敌人技能选择仍是“取第一个技能直接执行”
2. `SCOPE_PART` 技能的目标部位不是随机有效部位，而是固定取候选列表第一个
3. 敌人缺少独立行为层
4. 敌人模板缺少结构化 AI 数据
5. 行为逻辑与战斗执行逻辑耦合过高

---

## 3. 本次重构范围

### 3.1 第一阶段只实现的内容

第一阶段仅实现一个正式行为模式：

- `random_part_attacker`

第一阶段仅支持以下能力：

1. 从 `basicAttack` 技能池中选择技能
2. 若技能目标为 `SCOPE_PART`：
   - 若技能已固定部位，则直接使用固定部位
   - 否则从玩家有效部位中等概率随机选择 1 个
3. 生成标准化敌人 action，接入现有时间线与执行系统

### 3.2 第一阶段明确不做的内容

为了避免过度设计，以下内容暂不实现：

- 行为树
- 多阶段 Boss 策略
- 技能评分器
- 权重随机系统
- 仇恨系统
- 条件组合器
- 技能链 / 连招系统
- 自适应博弈型 AI

---

## 4. 设计原则

### 4.1 行为层与技能层解耦

- 技能层负责定义“技能能做什么”
- 行为层负责定义“敌人何时做、对谁做、打哪里”

行为层不得直接依赖“技能候选列表第一个部位”来代替真实的部位决策。

### 4.2 高内聚、低耦合

敌人 AI 决策逻辑应从 `CoreEngine` 中拆出，形成独立模块。

- `CoreEngine` 负责调度与执行
- AI 模块负责决策
- 目标解析与部位选择规则独立封装

### 4.3 严格数据驱动

敌人行为应完全由敌人模板中的 AI 数据驱动。

若关键字段缺失，则直接报错，不做回退。

### 4.4 与现有战斗规则保持一致

普通敌人的默认普通攻击应优先满足：

- `DMG_ARMOR`
- `SCOPE_PART`
- 先伤护甲，再将溢出结算到 HP

---

## 5. 第一阶段行为模式

## 5.1 `random_part_attacker`

适用对象：

- 普通小怪
- 杂兵
- 基础近战敌人
- 当前测试阶段的大多数敌人

行为规则：

1. 从 `basicAttack` 技能池中选择技能
2. 解析技能目标类型
3. 若目标为 `SCOPE_ENTITY`，则直接以玩家为目标
4. 若目标为 `SCOPE_PART`：
   - 若技能本身已固定部位，则直接使用固定部位
   - 否则从玩家有效部位中等概率随机选择 1 个
5. 生成敌人本回合 action

第一阶段中，`random_part_attacker` 作为普通敌人的默认行为模型。

---

## 6. 玩家有效部位选择规则

敌人选择玩家部位时，只能从以下标准集合中选择：

- `head`
- `chest`
- `abdomen`
- `arm`
- `leg`

并且必须满足：

- `max > 0`

### 6.1 第一阶段随机规则

对于 `random_part_attacker`：

1. 收集所有 `max > 0` 的玩家部位
2. 从中等概率随机选择 1 个
3. 作为当前技能的 `bodyPart`

### 6.2 第一阶段不额外引入的条件

当前阶段暂不加入以下筛选：

- 部位权重
- 仇恨偏好
- 伤害收益评估
- 破甲优先级
- 残废部位偏好

这些能力保留给后续行为模式扩展。

---

## 7. 固定部位技能规则

当技能本身已声明固定部位时，行为层不再随机部位。

### 7.1 可视为固定部位的情况

满足以下任一条件时，可视为技能固定部位：

1. `target.selection.selectedParts.length === 1`
2. action effect 中存在单一固定 `partOverride`

### 7.2 不能视为固定部位的情况

若技能只是配置了：

- `candidateParts = [...]`
- `selectedParts = []`

则不能将其视为固定部位技能。

此时应由行为层负责选择真实攻击部位。

---

## 8. 技能选择规则

### 8.1 第一阶段技能池

第一阶段仅使用：

- `basicAttack`

### 8.2 第一阶段选择策略

为保证实现简单、结果稳定，第一阶段采用顺序选择：

1. 从 `basicAttack` 中按数组顺序遍历
2. 取第一个合法可用技能
3. 若技能池为空或无合法技能，直接报错

### 8.3 第一阶段不做随机技能选择

当前阶段不对技能池进行随机抽取，原因：

- 当前敌人数量少
- 测试需要稳定结果
- 顺序选择更容易调试
- 后续改为随机或权重选择的成本较低

---

## 9. 敌人模板数据结构

建议在敌人模板中增加 `ai` 字段：

```json
{
  "id": "goblin_01",
  "name": "哥布林战士",
  "stats": {},
  "bodyParts": {},
  "skills": ["skill_heavy_swing"],
  "ai": {
    "behaviorId": "random_part_attacker",
    "skillGroups": {
      "basicAttack": ["skill_heavy_swing"],
      "breaker": [],
      "defense": [],
      "finisher": [],
      "utility": []
    }
  }
}
```

### 9.1 字段语义

- `skills`
  - 敌人拥有的技能列表
- `ai.behaviorId`
  - 敌人当前行为模式 ID
- `ai.skillGroups.basicAttack`
  - 普通攻击技能池
- `ai.skillGroups.breaker`
  - 预留：破甲技能池
- `ai.skillGroups.defense`
  - 预留：防御技能池
- `ai.skillGroups.finisher`
  - 预留：收割技能池
- `ai.skillGroups.utility`
  - 预留：功能技能池

### 9.2 第一阶段数据约束

对于进入战斗的普通敌人，应满足：

1. 必须存在 `ai`
2. 必须存在 `ai.behaviorId`
3. 必须存在 `ai.skillGroups.basicAttack`
4. `basicAttack` 不能为空
5. 技能 ID 必须存在于技能库中
6. 若行为需要部位目标，则必须能解析出合法部位

缺失任一关键数据时，直接报错。

---

## 10. 模块划分方案

为保持高内聚、低耦合，建议将敌人 AI 决策从 `CoreEngine` 中拆出。

## 10.1 `EnemyAiPlanner`

职责：

- 作为敌人 AI 的统一决策入口
- 根据敌人模板、玩家数据、战斗上下文生成标准 action

输入：

- `enemy`
- `player`
- `battleState`
- `skillRepo`

输出：

```json
{
  "source": "ENEMY",
  "sourceId": "goblin_01",
  "skillId": "skill_heavy_swing",
  "targetId": "player_xxx",
  "bodyPart": "chest",
  "cost": 0,
  "speed": 10
}
```

## 10.2 `EnemyBehaviorResolver`

职责：

- 根据 `behaviorId` 分发到具体行为实现

第一阶段仅支持：

- `random_part_attacker`

后续可扩展：

- `armor_breaker`
- `finisher`
- `defensive`
- `tactical`

## 10.3 `EnemyTargetingService`

职责：

- 解析技能目标类型
- 获取玩家有效部位集合
- 判断技能是否为固定部位技能
- 为行为层提供统一的目标选择能力

该模块不参与时间线、不参与伤害执行，只负责目标决策规则。

---

## 11. 与 `CoreEngine` 的接入方式

当前敌人 action 的生成逻辑写在 `CoreEngine` 中，并直接取：

- `enemy.skills[0]`
- 第一个候选部位

本次重构后，应改为：

1. `CoreEngine` 遍历存活敌人
2. 对每个敌人调用 `EnemyAiPlanner.plan(enemy, context)`
3. 由 `EnemyAiPlanner` 返回标准化 action
4. `CoreEngine` 将 action 写入 `enemySkillQueue`
5. 时间线系统统一处理玩家与敌人的行动顺序

### 11.1 `CoreEngine` 保留的职责

- 战斗流程调度
- 行动队列管理
- 时间线装载
- 技能执行
- 伤害结算

### 11.2 `CoreEngine` 不再承担的职责

- 直接决定敌人用哪个技能
- 直接决定敌人打哪个部位
- 直接写死敌人行为逻辑

---

## 12. 第一阶段的错误处理规则

为了避免继续堆叠临时兼容逻辑，以下情况必须直接报错：

1. 敌人缺少 `ai`
2. 敌人缺少 `behaviorId`
3. 行为模式未实现
4. `basicAttack` 技能池为空
5. 技能 ID 不存在
6. 目标主体无法解析
7. `SCOPE_PART` 技能无法解析出合法部位
8. 玩家不存在任何有效部位

不允许：

- 回退到 `enemy.skills[0]`
- 回退到默认 `chest`
- 回退到任意部位
- 静默跳过行为问题

---

## 13. 第一阶段测试目标

第一阶段的目标不是让敌人“聪明”，而是让敌人：

- 行为符合设计
- 输出可验证
- 错误可暴露
- 后续可扩展

### 13.1 必测场景

#### 场景 1：普通单技能敌人

如：`goblin_01`

期望：

- 每回合从 `basicAttack` 选择技能
- 随机选择玩家一个有效部位
- 技能按 `DMG_ARMOR` 路径结算

#### 场景 2：玩家有无效部位

如：部分部位 `max = 0`

期望：

- 敌人不会选择这些部位

#### 场景 3：固定部位技能

期望：

- 行为层不再随机部位
- 直接使用技能固定部位

#### 场景 4：配置错误

如：

- 缺少 `behaviorId`
- `basicAttack` 为空
- 技能不存在
- 无法解析部位

期望：

- 直接报错

---

## 14. 后续扩展方向

本方案第一阶段虽简化，但数据结构与模块边界已预留扩展空间。

后续可在不破坏当前结构的前提下扩展：

### 14.1 新行为模式

- `armor_breaker`
- `finisher`
- `defensive`
- `tactical`

### 14.2 新技能选择器

- 顺序选择
- 随机选择
- 权重随机
- 条件优先级选择

### 14.3 新部位选择策略

- 仅选仍有护甲的部位
- 优先攻击易打穿部位
- 优先攻击破甲部位
- 带权重的随机选择

### 14.4 新战斗上下文判断

- 玩家 HP 阈值判断
- 玩家部位护甲分布判断
- 敌人自身护甲危机判断
- Buff / Debuff 状态判断

---

## 15. 实施顺序建议

建议按以下顺序实施：

1. 新增本设计文档并作为实现依据
2. 为敌人模板补齐 `ai` 结构
3. 校验现有敌人技能引用是否合法
4. 新增 AI 模块：
   - `EnemyAiPlanner`
   - `EnemyBehaviorResolver`
   - `EnemyTargetingService`
5. 修改 `CoreEngine` 的敌人 action 构建入口
6. 补充最小测试场景

---

## 16. 结论

本方案是一个面向当前测试需求的敌人 AI 重构 MVP 方案，目标是：

1. 先让普通敌人具备符合设计的基础交互能力
2. 避免继续使用“取第一个技能 + 打第一个部位”的临时逻辑
3. 将敌人行为决策从 `CoreEngine` 中拆出，建立清晰模块边界
4. 用最小实现成本换取后续扩展能力

当前阶段推荐正式落地的核心结论：

1. 第一阶段只实现 `random_part_attacker`
2. 普通敌人仅从 `basicAttack` 技能池中选技能
3. 普通敌人对 `SCOPE_PART` 技能应随机攻击玩家有效部位
4. 固定部位技能由技能本身决定，不再额外随机
5. 所有关键配置缺失直接报错，不做兼容性兜底

本文件作为后续敌人 AI MVP 实现的正式设计依据。
